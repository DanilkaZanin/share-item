# Фаза 3. Оценка функционального объёма и трудоёмкости проекта

## 1. Назначение фазы

Целью третьей фазы является оценка масштаба и сложности разработки сервиса шеринга предметов на основе формализованных методов оценки программных проектов. Полученные оценки используются для понимания трудоёмкости, сроков реализации и состава команды.

В рамках фазы применяются несколько независимых подходов, что позволяет повысить надёжность итоговых выводов.

---

## 2. Акторы и границы системы

### Акторы

- **Пользователь сервиса**  
  Зарегистрированный пользователь, который может добавлять предметы, просматривать каталог, загружать изображения и взаимодействовать с другими пользователями.

- **Администратор системы**  
  Пользователь с расширенными правами, осуществляющий контроль и модерацию данных.

- **Объектное хранилище (S3)**  
  Внешний сервис для хранения файлов, используемый для загрузки и получения изображений предметов.

---

### Граница системы

В границы системы входят:

- backend-приложение на Spring Boot;
- реляционная база данных;
- интеграция с S3-совместимым хранилищем;
- подсистема безопасности и управления доступом.

Клиентские приложения и внешние сервисы рассматриваются как источники запросов через API.

---

## 3. Use Cases системы

Для оценки функционального объёма проекта были сформированы основные пользовательские сценарии, включая позитивные и альтернативные варианты использования.

### Основные Use Cases

- регистрация и аутентификация пользователя;
- просмотр и редактирование профиля;
- добавление предмета с описанием и изображениями;
- загрузка и хранение файлов в объектном хранилище;
- просмотр каталога доступных предметов;
- обновление и удаление предметов владельцем;
- управление доступностью предметов;
- поиск и фильтрация предметов;
- разграничение прав доступа к ресурсам.

Сформированный набор Use Cases покрывает ключевую бизнес-логику сервиса и используется как основа для дальнейших расчётов.

---

## 4. Оценка размера по методу COSMIC Function Points

Для оценки функционального объёма проекта был применён метод **COSMIC Function Points**, ориентированный на анализ потоков данных между пользователем, системой и внешними сервисами.

В рамках анализа были выделены:

- входные операции (запросы API);
- выходные операции (ответы системы);
- операции чтения и записи данных;
- взаимодействие с внешним объектным хранилищем.

По результатам подсчёта функциональных процессов суммарный размер системы оценивается примерно в:

**~30–35 COSMIC Function Points**

---

## 5. Оценка размера по методу Use Case Points (UCP)

Дополнительно была выполнена оценка сложности проекта с использованием метода **Use Case Points**, основанного на количестве и сложности сценариев использования.

При расчёте учитывались:

- количество акторов;
- сложность Use Cases;
- технические и средовые факторы, включая безопасность и контейнеризацию.

Оценка по методу UCP дала следующий результат:

**~90–110 Use Case Points**

---

## 6. Экспертная оценка трудоёмкости

Для дополнительной валидации результатов была проведена экспертная оценка сложности проекта в **Story Points**.

Экспертная оценка учитывала:

- количество бизнес-сущностей;
- интеграцию с внешними сервисами;
- требования к безопасности;
- необходимость развёртывания в контейнеризированной среде.

По итогам обсуждения суммарная оценка проекта составила:

**~85–100 Story Points**

---

## 7. Сравнение и интерпретация результатов

Все применённые методы оценки дали сопоставимые результаты и указывают на один порядок величины сложности проекта. Это позволяет сделать вывод о корректности проведённой оценки.

Проект относится к категории **средних по масштабу backend-систем** с дополнительной сложностью за счёт интеграций и требований к безопасности.

---

## 8. Оценка сроков и команды

При следующих допущениях:

- 1 Story Point ≈ 10–12 человеко-часов;
- средняя скорость команды — 20–25 SP за спринт;

можно получить ориентировочные параметры проекта:

- общие трудозатраты: **~900–1200 человеко-часов**;
- продолжительность разработки: **4–5 спринтов**;
- срок реализации: **2–3 месяца**.

### Рекомендуемый состав команды:

- backend-разработчик (Spring Boot);
- инженер по инфраструктуре / DevOps (Docker, окружение);
- при необходимости — дополнительный backend-разработчик.

---

## 9. Выводы по фазе

В рамках третьей фазы была выполнена комплексная оценка функционального объёма и трудоёмкости сервиса шеринга предметов. Использование нескольких методов позволило получить устойчивые и согласованные результаты.

Полученные оценки подтверждают реализуемость проекта в заданных временных рамках и служат основой для детального календарно-ресурсного планирования в следующей фазе.
